BatchAdd 页面迁移与 Stepper 全宽调整 — 2026-05-30

概述
- 将原本对话框式的批量新增改为独立页面（路由 `/table/batch-add`），并使向导在页面中以全宽展示，解决第二步表格被挤断行的问题。

关键改动（文件）
- frontend/src/views/table.vue -> frontend/src/views/table/index.vue
  - 将 table 视图移入 `views/table` 目录，作为 `/table` 路由入口。
- frontend/src/views/table/batch-add.vue （新增）
  - 页面壳（标题/返回），内嵌 BatchAdd 组件。
- frontend/src/components/table/BatchAdd.vue（修改）
  - 移除模态模式与尺寸调整（resize / maximize）逻辑，改为页面模式。
  - 提交成功后新增第4步状态页（展示已添加数量 + 后端初始化提示 + spinner + 操作按钮）。
  - 调整样式：移除 card 最大宽度、缩小左右内边距，使 stepper 填满父容器宽度和可用空间。
  - 逻辑微调（savedCount、applyTargetTableTemplate 等）。
- frontend/src/views/table/index.vue（修改）
  - 删除以往打开 BatchAdd 模态的动态 import / componentMap 条目。
  - 把原先 `openDialog('BatchAdd', ...)` 的入口替换为路由跳转 `router.push('/table/batch-add')`（新增 `goToBatchAdd`）。

动机（Why）
- 批量新增向导流程较长且高频，模态对话框受限导致信息拥挤与操作不便；页面模式支持更多空间、更好导航与书签；同时解决了 stepper 左右留白过多导致内容换行的问题。

验证状态（Validation）
- 已在工作区安装依赖并执行构建：`yarn --cwd frontend build` → 构建成功（vite build 通过）。
- 无 TypeScript 或打包错误。
- 推荐的手动验证步骤：
  1. 启动 dev：`VITE_API_HOST=http://localhost:50601 yarn --cwd frontend dev`
  2. 访问 `/table`，点击“新增表”或 EmptyState 的“新增采集表” → 跳转到 `/table/batch-add`。
  3. 完整走一遍向导（Step1 选源+库、Step2 加载并选择表、Step3 配置并提交），确认第2步表格在一行能完整展示列内容。
  4. 提交成功后应进入第4步状态页，点击“返回采集表”应带 `?refresh=` 并触发表格刷新。

影响与后续（Impact / Follow-ups）
- 影响范围：仅前端 UI 变更，后端接口未修改。
- 风险：移除了模态入口，需确认项目中无其它隐藏逻辑依赖旧的模态方式（已在 table 视图替换已发现的大多数调用）。
- 后续建议：
  - 若需实时显示后端初始化进度，建议后端在 batchSave 返回 jobId 并提供状态接口，前端在第4步轮询展示进度。
  - 可选将第4步改为使用 Vuetify 的 `v-empty-state` 组件（当前使用自定义卡片回退实现以兼容性更稳）。
